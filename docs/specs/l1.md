# L1 — High Level Requirements

## 1. Purpose

QuorumQ is a mobile-first, responsive web application that helps software teams
democratically decide where to have lunch. Team members can suggest restaurants,
vote on suggestions, and leave review comments. The app must deliver a
spectacular, polished user interface and be engineered using
Acceptance Test Driven Development (ATDD) with end-to-end tests implemented in
Playwright following the Page Object Model.

## 2. Scope

This document defines the high-level capabilities of QuorumQ. Each requirement
below is assigned a stable identifier (`L1-xx`) that detailed requirements in
`l2.md` trace back to.

## 3. High Level Requirements

### L1-01 — Team Membership
The system shall allow users to create and join teams so that lunch decisions
are scoped to a team of coworkers.

### L1-02 — User Authentication
The system shall authenticate users and associate their actions (suggestions,
votes, comments) with their identity.

### L1-03 — Restaurant Suggestions
The system shall allow authenticated team members to suggest restaurants as
lunch candidates for a given lunch session.

### L1-04 — Lunch Sessions
The system shall allow a team to create a time-bounded lunch session during
which suggestions, voting, and commenting occur.

### L1-05 — Voting
The system shall allow team members to vote on restaurant suggestions during an
active lunch session and shall determine a winning restaurant from the votes.

### L1-06 — Comments and Reviews
The system shall allow team members to leave comments and reviews on
restaurants and on individual suggestions within a lunch session.

### L1-07 — Real-time Collaboration
The system shall reflect suggestions, votes, and comments to all team members
in near real time so that the team can decide quickly.

### L1-08 — Mobile-First Responsive UI
The system shall provide a mobile-first, fully responsive user interface that
works well on phones, tablets, and desktop browsers.

### L1-09 — Spectacular UI Design
The system shall deliver a visually spectacular, polished, accessible UI with a
coherent design system, fluid animations, and delightful interactions.

### L1-10 — Accessibility
The system shall conform to WCAG 2.1 AA accessibility guidelines.

### L1-11 — Performance
The system shall load quickly and remain responsive on typical mobile networks
and devices.

### L1-12 — End-to-End Testing with Playwright
The system shall be verified by end-to-end tests written in Playwright using
the Page Object Model pattern.

### L1-13 — Acceptance Test Driven Development
The system shall be implemented using Acceptance Test Driven Development, where
each user-visible feature begins life as a failing acceptance test derived from
the acceptance criteria in `l2.md`.

### L1-14 — History and Transparency
The system shall record the history of past lunch sessions, winning
restaurants, and participation so teams can review previous decisions.

### L1-15 — Notifications
The system shall notify team members of relevant lunch events (session
started, voting closing soon, winner decided) without being intrusive.

### L1-16 — Data Privacy and Security
The system shall protect user data and restrict access to team data to members
of that team.

### L1-17 — Technology Stack
The system shall be implemented as a .NET backend Web API with an Angular
single-page application front end that uses Angular Material for all UI
components.

### L1-18 — Radical Simplicity
The implementation shall be radically simple: it shall deliver the
requirements in this document with the smallest reasonable amount of code,
the fewest layers, and no speculative abstractions or frameworks-within-the-
framework.
